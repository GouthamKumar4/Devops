🎯 Introduction: The Door to Your Cloud Computer

Imagine this scenario:

You have a computer sitting in Microsoft's data center. You're sitting at home. You need to access it securely across the internet. How do you prove you're allowed in?

That's what VM authentication is all about – proving you are who you say you are before getting access to a virtual server.

Why This Matters

✅

Good Authentication

  • Only authorized users get in
  • Audit trail of who accessed what
  • Can revoke access instantly
  • Protects sensitive data
❌

Bad Authentication

  • Unauthorized access
  • Data breaches
  • Ransomware attacks
  • Compliance violations

🔑 SSH Keys Explained (Layman Terms)

The Lock and Key Analogy

Imagine two special keys:

  • 🔓 Public Key = A special lock you put on your server
  • 🔐 Private Key = The only key that can open that lock (stays with you)

Anyone can have the public key (the lock), but only someone with the matching private key can open it.

The Magic of Public-Key Cryptography

🔐

Private Key

Stays on YOUR computer

⚠️ NEVER share this

➡️
🔓

Public Key

Lives on the VM

✅ Safe to share

How SSH Login Works (Step by Step)

👨‍💻 You type: ssh azureuser@myvm.com
⬇️
🖥️ VM says: "Prove you own the private key!"
⬇️
🔐 Your computer signs a challenge with private key
⬇️
🔓 VM verifies signature with public key
⬇️
✅ Access Granted!
The Beautiful Security Trick: Your private key NEVER leaves your computer. The VM never sees it. You prove ownership without revealing the secret.

Creating SSH Keys

# Generate new SSH key pair
ssh-keygen -t rsa -b 4096 -C "your_email@example.com"
# This creates two files:
~/.ssh/id_rsa # Private key (keep secret!)
~/.ssh/id_rsa.pub # Public key (share this)

SSH Key Formats

Format Command Key Size Security Usage
RSA ssh-keygen -t rsa -b 4096 2048-4096 bits High Most common, widely supported
Ed25519 ssh-keygen -t ed25519 256 bits Very High Modern, faster, smaller keys
ECDSA ssh-keygen -t ecdsa -b 521 256-521 bits High Elliptic curve, good performance
DSA ssh-keygen -t dsa 1024 bits Deprecated ⚠️ Don't use (weak)

Azure Recommendation

Azure supports all modern SSH key formats. Recommended order:

  1. Ed25519 - If all your systems support it (best choice)
  2. RSA 4096 - Maximum compatibility with high security
  3. ECDSA - Good alternative if Ed25519 not available

Where Keys Live on Linux

# Your private key (on your laptop)
~/.ssh/id_rsa
# Allowed public keys (on the VM)
/home/azureuser/.ssh/authorized_keys
# Example content of authorized_keys:
ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAACAQC... user@laptop

🆘 What Happens When SSH Keys Are Lost?

The Problem

Imagine you lost your house key, but a copy is still in the lock. The house "trusts" that key, but you no longer have it. You need a way to add a NEW key to the lock.

With VMs, the server has a public key it trusts, but your private key is gone. You can't authenticate anymore. Now what?

Important: The private key never existed on the VM, so there's no way to "recover" it. You must inject a NEW public key using an alternative access method.

Recovery Methods

Method 1: Azure Portal Reset (Easiest) ▼

How Azure Portal Reset Works

Navigate to VM in Azure Portal
⬇️
Click "Reset Password"
⬇️
Choose "Reset SSH public key"
⬇️
Paste NEW public key
⬇️
Azure VM Agent writes to ~/.ssh/authorized_keys
⬇️
✅ Can now login with new private key!
# Step 1: Generate new key pair on your laptop
ssh-keygen -t ed25519 -f ~/.ssh/azure_recovery
# Step 2: Copy the public key
cat ~/.ssh/azure_recovery.pub
# Step 3: Paste into Azure Portal reset form
# Step 4: Try login
ssh -i ~/.ssh/azure_recovery azureuser@myvm.azure.com
Method 2: Azure CLI Reset ▼
# Reset SSH key via Azure CLI
az vm user update \
--resource-group myResourceGroup \
--name myVM \
--username azureuser \
--ssh-key-value "$(cat ~/.ssh/newkey.pub)"

This does the same thing as portal method, but through CLI. The Azure VM Agent receives the command and updates the authorized_keys file.

Method 3: Serial Console Access (Break Glass) ▼
Serial Console is like plugging a keyboard and monitor directly into a physical server. It bypasses the network entirely and gives you direct access to the Linux terminal.

Enable Serial Console

  1. Go to VM in Azure Portal
  2. Select "Serial Console" from left menu
  3. Login with username/password (if configured)
  4. Manually edit authorized_keys file
# Once in serial console:
cd ~/.ssh
nano authorized_keys
# Add your new public key, save and exit
Serial Console requires that you enabled password authentication OR boot diagnostics. If both SSH keys are lost AND password auth is disabled, this won't work.
Method 4: Disk Surgery (Advanced) ▼
Think of this like removing a hard drive from a broken computer, plugging it into another computer, fixing the files, then putting it back. It's invasive but works when nothing else does.
1

Stop the VM

Deallocate the broken VM in Azure Portal

2

Detach OS Disk

Remove the OS disk from the VM

3

Attach to Recovery VM

Attach the disk to a different working VM as a data disk

4

Mount and Fix

Mount the disk, navigate to authorized_keys, add new key

5

Reattach and Start

Unmount, detach, reattach to original VM, start VM

# On the recovery VM, after attaching disk:
sudo mkdir /mnt/broken-vm
sudo mount /dev/sdc1 /mnt/broken-vm
# Edit the authorized_keys file
sudo nano /mnt/broken-vm/home/azureuser/.ssh/authorized_keys
# Add new public key, save, unmount
sudo umount /mnt/broken-vm
Method 5: Rebuild from Image (Cloud-Native) ▼

The Modern Philosophy: Cattle, Not Pets

Instead of trying to "fix" a broken server, just replace it with a fresh one from a template.

❌ VM with lost keys
⬇️
🗑️ Terminate VM
⬇️
📦 Create new VM from golden image
⬇️
🔑 Inject new SSH keys during creation
⬇️
✅ Fresh VM, properly configured

Why this is better:

  • Faster than troubleshooting
  • Guaranteed clean state
  • No accumulated configuration drift
  • Infrastructure as Code approach
This only works if you've properly designed for immutable infrastructure – data on external volumes, configuration in code, etc.

👤 Identity-Based SSH (Modern Approach)

The Problem with SSH Keys at Scale

Imagine a company with:

  • 500 employees
  • 1000 virtual machines
  • Each person needs access to ~20 VMs

That's potentially 10,000 SSH keys to manage! When someone leaves the company, you need to remove their key from all VMs. When someone's laptop is stolen, panic ensues.

Solution: Use corporate identity instead of managing individual keys.

How Identity-Based SSH Works

👨‍💼 Employee
→
🏢 Azure AD / Entra ID
→
🎫 OAuth Token
→
🖥️ VM

The Flow in Detail

1

Employee Logs Into Azure

az login

Authenticates via browser with MFA (multi-factor authentication)

2

Azure AD Issues Identity Token

Token contains: user identity, roles, permissions, expiration time

3

Request SSH Access

az ssh vm --name myVM --resource-group myRG

Azure CLI sends token to VM

4

VM Validates Token

Azure VM Login Extension verifies:

  • Token signature (is it really from Azure AD?)
  • Token expiration (is it still valid?)
  • User permissions (does RBAC allow this?)
5

Dynamic User Creation

VM dynamically maps identity to Linux user based on role:

  • Virtual Machine User Login → regular user
  • Virtual Machine Administrator Login → sudo access

Azure-Specific Implementation

Prerequisites:
  1. Install Azure AD Login Extension on VM
  2. Assign RBAC roles to users
  3. Enable Managed Identity on VM
# Install AAD Login Extension
az vm extension set \
--publisher Microsoft.Azure.ActiveDirectory \
--name AADSSHLoginForLinux \
--resource-group myRG \
--vm-name myVM
# Assign RBAC role to user
az role assignment create \
--role "Virtual Machine Administrator Login" \
--assignee user@company.com \
--scope /subscriptions/{sub-id}/resourceGroups/myRG/providers/Microsoft.Compute/virtualMachines/myVM

Benefits of Identity-Based SSH

🎯

Centralized Control

All access controlled from single identity system. Disable user in Azure AD = instant loss of all VM access.

📋

Audit Trail

Every login tied to real identity. Know exactly who accessed what and when.

🔐

MFA Enforcement

Leverage conditional access policies. Require MFA, device compliance, IP restrictions.

🚀

Zero Key Management

No SSH keys to distribute, rotate, or revoke. Identity system handles everything.

⏱️ Ephemeral SSH Certificates (Cloud-Native)

The Boarding Pass Analogy

Think about airport security:

  • Old way (SSH Keys): You get a permanent "VIP pass" that works forever. If someone steals it, they have unlimited access.
  • New way (Certificates): You get a temporary boarding pass valid for ONE flight. After that, it's useless.

Ephemeral certificates are like those temporary boarding passes – they expire quickly, so even if stolen, they become worthless within minutes.

How SSH Certificates Differ from Keys

Aspect SSH Keys SSH Certificates
Lifespan Forever (until manually removed) Minutes to hours (auto-expire)
Management Each VM stores authorized_keys VMs trust one Certificate Authority
Revocation Must manually remove from all VMs Automatic (certificate expires)
Stolen Credential Risk High (permanent access) Low (access expires quickly)
Best For Small deployments, personal use Enterprise, large-scale, high-security

Certificate-Based SSH Architecture

👨‍💻 Engineer requests access
⬇️
🏢 Authenticate to Identity Provider (SSO + MFA)
⬇️
🎫 Certificate Authority signs temporary certificate
(valid for 5-15 minutes)
⬇️
🔐 Certificate contains: identity, permissions, expiration
⬇️
🖥️ Present certificate to VM
⬇️
✅ VM validates CA signature + checks expiration
⬇️
🚪 Access granted (for remaining time)

How Certificate Validation Works

The Trust Model:

Instead of each VM trusting 100 different user keys, each VM trusts ONE Certificate Authority (CA).

Think of it like a government issuing passports. Border control doesn't need to know every citizen – they just trust any passport signed by their government.

Old Way: VMs Trust Individual Keys

🖥️ VM

Trusts:

🔑 Alice
🔑 Bob
🔑 Carol
🔑 Dave
🔑 ...100 more

New Way: VMs Trust One CA

🖥️ VM

Trusts:

🏢 CA Only
→
🏢 CA

Signs certificates for:

✅ Alice (5 min)
✅ Bob (10 min)
✅ Carol (5 min)

Certificate Lifecycle

1

Request Time: 0 minutes

Engineer authenticates with identity provider, CA issues certificate valid for 10 minutes

2

Minutes 0-10: Certificate Valid

Engineer can SSH to any VM. Each VM validates the certificate signature and checks expiration time.

3

Minute 10: Certificate Expires

Certificate becomes invalid. Access automatically denied. No manual revocation needed.

4

Need More Access?

Request new certificate (re-authenticate if needed). Fresh 10-minute window.

Azure-Specific Authentication Methods

1. Azure Portal SSH (Browser-Based) ▼

Newest method – SSH directly from browser without any client software!

How it works:

  1. Navigate to VM in Azure Portal
  2. Click "Connect" → "SSH"
  3. Choose authentication (key or Azure AD)
  4. Terminal opens in browser
No need to install SSH client, manage keys, or know IP addresses. Everything happens through the browser. Azure handles authentication and routing.
2. Azure CLI SSH ▼
# SSH using Azure AD authentication
az ssh vm --name myVM --resource-group myRG
# SSH using local SSH key
az ssh vm --name myVM --resource-group myRG --local-user azureuser
# SSH through Azure Bastion
az network bastion ssh --name myBastion --resource-group myRG \
--target-resource-id /subscriptions/.../myVM --auth-type AAD
3. Azure Bastion (Zero Public IP) ▼

What is Azure Bastion?

Imagine a heavily guarded entry gate to your private network. You connect to the gate (which is public), and the gate provides secure access to your private VMs (which have no public IP).

Without Bastion (Risky)

🌐 Internet
⬇️
🖥️ VM with Public IP
(Directly exposed)

With Bastion (Secure)

🌐 Internet
⬇️
🛡️ Azure Bastion
(Hardened gateway)
⬇️
🔒 Private VNet
⬇️
🖥️ VM (No public IP)

Bastion Features

  • VMs don't need public IPs
  • No need to manage NSG rules for SSH/RDP
  • SSL/TLS encrypted session through port 443
  • Protection against port scanning
  • Integrated with Azure AD
  • Session recording available
4. Just-In-Time (JIT) VM Access ▼

Imagine SSH port is normally LOCKED. When you need access, you request permission. The system temporarily unlocks it for you (e.g., 3 hours), then automatically locks it again.

JIT Access Flow

Engineer needs VM access
⬇️
Request JIT access (Security Center)
⬇️
Azure verifies RBAC permissions
⬇️
NSG rule added: Allow SSH from your IP for 3 hours
⬇️
SSH access granted (time-limited)
⬇️
After 3 hours: NSG rule automatically removed

Enabling JIT

Requires Microsoft Defender for Cloud (formerly Security Center)

# Request JIT access via CLI
az security jit-policy create \
--location eastus \
--name JIT-Policy \
--resource-group myRG \
--virtual-machines "myVM"
5. Run Command (No SSH Needed) ▼
Sometimes you don't need a full SSH session – you just want to run ONE command. Run Command lets you do that through Azure, bypassing SSH entirely.
# Execute single command on VM
az vm run-command invoke \
--resource-group myRG \
--name myVM \
--command-id RunShellScript \
--scripts "cat /etc/os-release"
Run Command uses the Azure VM Agent to execute commands. Useful for:
  • Emergency access when SSH is broken
  • Automation without SSH
  • Quick diagnostics

Azure RBAC Roles for VM Access

Role Access Level Use Case
Virtual Machine User Login Regular user SSH access (no sudo) Developers who need to check logs, run apps
Virtual Machine Administrator Login Admin access with sudo privileges System administrators, DevOps engineers
Virtual Machine Contributor Manage VM (start/stop/config), but not access inside Platform teams managing infrastructure
Reader View VM details, no changes or access Auditors, management visibility

🤖 Azure VM Agent Explained

The Secret Helper Inside Your VM

Imagine you have a smart assistant living inside your VM that:

  • Talks to Azure control plane
  • Installs software you request from Azure Portal
  • Resets passwords when you're locked out
  • Reports health status
  • Executes commands sent from Azure

That's the Azure VM Agent – a small software service running inside every Azure VM.

VM Agent Architecture

☁️ Azure Control Plane
(Portal, CLI, ARM API)
⬇️ Communication via wire protocol ⬇️
🤖 VM Agent
(waagent / WindowsAzureGuestAgent)
⬇️ Controls ⬇️
🖥️ VM Components
(OS, files, extensions, networking)

What VM Agent Does

🔧

Extension Management

Installs and manages VM extensions (antivirus, monitoring, backup agents, etc.)

🔐

Credential Management

Handles SSH key injection during VM creation and password resets

💚

Health Reporting

Reports VM status, extension status, and diagnostics to Azure

📦

Provisioning

Sets up VM during first boot (hostname, networking, user accounts)

Extension Purpose Example Use

VM Agent and Authentication

How VM Agent Enables Recovery

When you reset SSH keys through Azure Portal, here's what happens:

1

You submit new SSH key through Azure Portal

2

Azure sends command to VM Agent inside the VM

3

VM Agent (running as root) writes to ~/.ssh/authorized_keys

4

VM Agent reports success back to Azure

5

You can now SSH with new key!

Checking VM Agent Status

# Check if VM Agent is running (Linux)
systemctl status walinuxagent
# Check agent version
waagent --version
# View agent logs
tail -f /var/log/waagent.log
# Check from Azure CLI
az vm get-instance-view --resource-group myRG --name myVM \
--query "instanceView.vmAgent"
What if VM Agent is broken?

If VM Agent fails:

  • You can't reset passwords from Azure Portal
  • Extensions won't install/update
  • Run Command won't work
  • You'll need Serial Console or disk surgery for recovery

Always monitor VM Agent health in production!

📦 VM Images and Golden Images

The Cookie Cutter Analogy

Imagine baking cookies:

  • Bad way: Manually shape each cookie by hand (slow, inconsistent)
  • Good way: Use a cookie cutter – make a perfect template, stamp out identical cookies

VM images are like cookie cutters. You create ONE perfect VM configuration (the "golden image"), then create hundreds of identical VMs from it instantly.

Image Types in Azure

🏪

Marketplace Images

Public

Pre-built images from Microsoft and partners

Examples: Ubuntu 22.04, Windows Server 2022, RHEL

Use when: Starting fresh, standard configurations

🎖️

Golden Images

Custom

Your customized, pre-configured images

Contains: Your apps, configs, security hardening

Use when: Deploying 10+ similar VMs

📸

Snapshots

Backup

Point-in-time copy of a disk

Contains: Exact state including data

Use when: Backups, disaster recovery

🎨

Azure Compute Gallery

Enterprise

Centralized image repository with versioning

Contains: Managed golden images across regions

Use when: Multi-region, multi-team deployments

Creating a Golden Image

1

Create Template VM

Deploy VM from marketplace image, configure it perfectly:

  • Install required software
  • Apply security hardening
  • Configure monitoring agents
  • Set up networking
  • Remove sensitive data
2

Generalize the VM

Remove machine-specific data (hostname, SSH keys, etc.)

# Linux: Deprovision VM
sudo waagent -deprovision+user -force
# This removes:
# - SSH host keys
# - User accounts created by waagent
# - Cached network configs
3

Deallocate and Generalize

az vm deallocate --resource-group myRG --name templateVM
az vm generalize --resource-group myRG --name templateVM
4

Create Image

az image create \
--resource-group myRG \
--name myGoldenImage \
--source templateVM
5

Deploy VMs from Image

az vm create \
--resource-group prodRG \
--name prodVM01 \
--image myGoldenImage \
--ssh-key-value "$(cat ~/.ssh/id_rsa.pub)"

🎯 Authentication Best Practices

Security Maturity Model

1

Level 1: Basic (Small Projects)

Security: Medium
  • SSH keys instead of passwords
  • Ed25519 or RSA 4096 keys
  • No public SSH without firewall rules
2

Level 2: Intermediate (Team Projects)

Security: High
  • Azure Bastion for private network access
  • Just-In-Time VM access
  • Key rotation policy (90 days)
  • Audit logging enabled
  • VMs in private subnets
3

Level 3: Advanced (Enterprise)

Security: Very High
  • Identity-based SSH (Azure AD)
  • Ephemeral certificates (5-15 min lifetime)
  • Zero public IPs (all via Bastion/VPN)
  • Conditional access policies + MFA
  • Session recording
  • Immutable infrastructure (replace, don't fix)

Quick Decision Guide

Your Situation Recommended Approach
Personal learning project (1-5 VMs) SSH keys + Public IP with NSG restrictions
Small startup (5-20 VMs) SSH keys + Azure Bastion + Private VNet
Growing company (20-100 VMs) Azure AD SSH + Bastion + JIT Access
Enterprise (100+ VMs) Ephemeral certificates + Zero Trust + Immutable infra
High-security / Compliance Certificate-based + Session recording + No public access

Common Mistakes to Avoid

🚨

Exposing SSH to 0.0.0.0/0

Allowing SSH from anywhere = constant brute-force attacks

Fix: Use Azure Bastion or restrict to known IPs

🔑

Sharing Private Keys

Emailing SSH private keys defeats the whole purpose

Fix: Each person generates their own key pair

📅

Never Rotating Keys

SSH keys used for years = high risk if compromised

Fix: Rotate keys quarterly or use certificates

💾

Keys in Source Control

Accidentally committing private keys to Git repos

Fix: Use .gitignore, never commit .ssh/

The Modern Cloud VM Access Philosophy

From Permanent Access to Temporary Access

Old mindset: "I need permanent access to this server"

New mindset: "I need temporary access when needed, for specific tasks"

Old infrastructure: Servers are pets (named, cared for, fixed when broken)

New infrastructure: Servers are cattle (numbered, replaceable, rebuilt from templates)

The goal isn't just better authentication – it's designing systems where you rarely need to SSH at all.
