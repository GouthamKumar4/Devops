📖0. The Story — Why Was Git Created?

🎬 Imagine This Scene...

It's 2005. Linus Torvalds (creator of Linux) is managing the world's largest open-source project. Thousands of developers across the globe are sending code patches by email. The tool they were using (BitKeeper) just revoked their free license.

Linus looked at every existing version control system and said: "They're all too slow, too centralized, or too stupid."

So he built Git. In 2 weeks. From scratch. 🤯

📦0.3 Git's Weakness — Large Files & What To Do

🚨 Git Was Built for TEXT, Not Binary Files!

Remember: Git stores complete snapshots. Every time you change a file, Git stores a completely new blob. For text files (code), this works amazingly because:

  • Text files are small (a few KB)
  • Git can compress them efficiently
  • Git can calculate diffs between text versions

But for binary files (videos, images, datasets, ML models, compiled executables)...

💣 The Large File Problem — Visualized

❌ Video File (demo.mp4 = 500 MB)

v3: 500 MB (entire new blob again!)
Total in .git: 1.5 GB!!! 💥
Can't diff binary → stores 3 full copies

⚠️ What Happens When People DO Put Large Files in Git

  • Clone takes forever — every person downloads ALL versions of ALL large files
  • Repo size explodes — a 10 MB project becomes 5 GB with a few video versions
  • Push/pull is slow — transferring gigabytes over network
  • GitHub rejects files > 100 MB — and warns at 50 MB
  • Can't undo it easily — even after deleting the file, the blob is still in history!

✅ Solutions — How People Handle Large Files

📦 Git LFS (Large File Storage)

Replaces large files with tiny text pointers in Git. The actual files are stored on a separate server.

# Install and track large files git lfs install git lfs track "*.mp4" git lfs track "*.psd" git lfs track "*.zip" # Now git works normally! git add video.mp4 git commit -m "Add demo video" # Git stores a pointer, LFS stores the file

🛡️ .gitignore (Prevention)

Simply don't track files that don't belong in Git.

# .gitignore file *.mp4 *.zip *.exe *.dll node_modules/ *.log build/ dist/ __pycache__/ *.pyc

Best practice: Only track source code and config. Build artifacts, dependencies, and media go elsewhere.

💡 But People STILL Use Git for Everything — Why?

Because for 99% of software development work, files are text-based code (a few KB each). Git handles this perfectly. The large file problem only affects specific workflows (game dev, ML/AI, media production). Even those teams use Git — they just add Git LFS on top.

The benefits of Git (speed, branching, distributed, free, ecosystem) massively outweigh the large file limitation.

🎯1. Fundamentals — What is Git?

Git is NOT a File System - It's a Content Tracker

Git doesn't store files. Git stores content. This is the most important concept to understand.

Core Principles

✅ Principle #1: Content-Addressable Storage

Every piece of data gets a unique SHA-1 hash based on its content.

echo "Hello Git" | git hash-object --stdin # Output: 8d0e41234f24b6da002d962a26c2495ea16a425f # Same content = Same hash ALWAYS echo "Hello Git" | git hash-object --stdin # Output: 8d0e41234f24b6da002d962a26c2495ea16a425f

Why this matters: Git automatically deduplicates data! If you have 100 files with the same content, Git stores it only once.

✅ Principle #2: Snapshots, Not Diffs

Unlike other version control systems (like SVN), Git doesn't store file differences. Every commit is a complete snapshot of your entire project.

❌ Other VCS (Stores Differences)
Version 1: Full file Version 2: +3 lines, -1 line Version 3: +5 lines, -2 lines

To get Version 3, must apply all diffs sequentially.

✅ Git (Stores Snapshots)
Commit 1: Complete snapshot Commit 2: Complete snapshot Commit 3: Complete snapshot

To get any version, just point to that commit!

✅ Principle #3: Everything is Local (Almost)

99% of Git operations happen on your local machine. No network needed!

  • Local operations: commit, branch, merge, log, diff, checkout
  • Network operations: push, pull, fetch, clone

⚠️ Common Misconception: "Git Tracks Changes"

FALSE! Git doesn't track changes. Git takes snapshots of content.

When you run git diff, Git is calculating the difference between two snapshots. The diff doesn't exist in storage - it's computed on the fly!

📦2. The Git Object Model

Git Has Only 4 Types of Objects

Everything in Git is built from these four object types:

📄 BLOB (Binary Large Object)

What it stores: File content ONLY

Does NOT store: Filename, directory, permissions

Think of it as: A file's data, nothing more

📁 TREE

What it stores: Directory structure

Contains: Pointers to blobs and other trees

Think of it as: A folder with files and subfolders

📌 COMMIT

What it stores: Snapshot metadata

Contains: Tree pointer, parent pointer, author, message

Think of it as: A save point in your project

🏷️ TAG (Annotated)

What it stores: Named reference + metadata

Contains: Commit pointer, tagger, date, message

Think of it as: A permanent bookmark with a note

⚡ Every Small Change = Entirely New Blob Object

🚨 THE MOST IMPORTANT RULE IN GIT

Even if you change ONE single character in a file, Git creates a completely NEW blob with a completely different SHA hash. The old blob still exists. Nothing is overwritten — ever.

Watch: One Line Added → Completely New SHA

Added one line "Welcome!"  ⟹  SHA went from b00001 to b00003 — NO similarity!

🧠 Why Does This Happen?

SHA-1 is a cryptographic hash. It uses the entire file content to calculate the hash. Change 1 byte → the math cascades into a completely different 40-character result.

# Try this yourself in your hello-app project: echo "Hello World!" | git hash-object --stdin # → b00001... (one hash) echo -e "Hello World!\nWelcome!" | git hash-object --stdin # → b00003... (completely different!) # Even adding a SPACE changes it: echo "Hello World! " | git hash-object --stdin # → yet another hash!

✅ Both Blobs Still Exist!

Git NEVER deletes or overwrites old blobs. Both versions live side by side in .git/objects/:

📄 Blob b00001
"Hello World!"
← still exists, used by Commit 1
📄 Blob b00003
"Hello World!\nWelcome!"
← new! used by Commit 2

Key Point: Each commit's tree points to whichever blob was current at that time. Old commits still point to old blobs. This is how you can always go back to any version!

Blob Object - Deep Dive

Example: Create a Blob Manually (using our hello.txt!)

1 Create file content
echo "Hello World!" > hello.txt

This creates a file in your working directory (not in Git yet!)

2 Create a blob from content
git hash-object -w hello.txt # Output: b00001... (your actual hash will differ) # -w means "write to .git/objects/"

What happened:

  • Git read "Hello World!"
  • Added header: "blob 12\0"
  • Calculated SHA-1 hash: b00001...
  • Compressed with zlib
  • Saved to: .git/objects/b0/0001...
3 Verify the blob exists
git cat-file -t b00001 # Output: blob git cat-file -p b00001 # Output: Hello World!

The blob is now permanently stored in Git's database!

🤔 Why Doesn't the Blob Store the Filename?

Because multiple files might have the same content!

# Create two files with identical content echo "Hello World!" > hello.txt echo "Hello World!" > greeting.txt # Both create the SAME blob! git hash-object hello.txt # Output: b00001... git hash-object greeting.txt # Output: b00001... (SAME hash!)

Result: Git stores "Hello World!" only once, saving space! The tree object remembers which filename maps to which blob.

🌿6. Branching and Merging

Branches Are Just Pointers!

This is the most important concept about Git branches. A branch is literally just a 40-character file containing a commit hash.

git branch - What Actually Happens

1 Create a branch
git branch feature

That's it! Git just created a file.

2 See what happened
cat .git/refs/heads/feature # c00002... cat .git/refs/heads/main # c00002... # Both branches point to the SAME commit!
3 Branches are cheap!
# A branch is just 41 bytes (40-char hash + newline) ls -lh .git/refs/heads/feature # 41 bytes # Creating 1000 branches = 41KB # That's why Git branches are so fast!

1 Switch branches

2 Git updates HEAD
cat .git/HEAD # Before: ref: refs/heads/main # After: ref: refs/heads/feature # HEAD now points to feature branch!
3 Git updates working directory
# Git reads the commit that feature points to # Updates working directory files to match that commit # If files need to change, Git modifies them

If you have uncommitted changes:

error: Your local changes would be overwritten by checkout. Please commit or stash them.

🔀7. Rebase vs Merge — Complete Guide

⚠️ THE GOLDEN RULE

NEVER rebase commits that exist outside your repository!

If other people have pulled your commits, DO NOT rebase them. You'll create duplicate commits and confuse everyone.

Merge - Combining Branches

What git merge Does

1 Initial state
A ← B ← C (main) \ D ← E (feature)
2 Merge feature into main
git checkout main git merge feature
3 Git creates merge commit
A ← B ← C ← M (main) \ / D ← E (feature) # M is merge commit with TWO parents: C and E
4 Examine merge commit
git cat-file -p HEAD tree xyz... parent C (commit on main) parent E (commit on feature) author John Doe committer John Doe Merge branch 'feature'

The merge commit has TWO parents! This preserves the branch history.

✅ When to Use Merge

  • Public branches: Branches that others have pulled
  • Preserve history: Want to see when branches were created/merged
  • Team collaboration: Multiple people working on same branch
  • Feature branches: Merging completed features into main

Rebase - Rewriting History

What git rebase Does

1 Initial state
A ← B ← C (main) \ D ← E (feature)
2 Rebase feature onto main
git checkout feature git rebase main
3 Git replays commits

Behind the scenes:

# Step 3a: Find common ancestor (B) # Step 3b: Take commits D and E # Step 3c: Temporarily save them # Step 3d: Reset feature branch to main (C) # Step 3e: Apply D on top of C → creates D' # Step 3f: Apply E on top of D' → creates E'
4 Result: Linear history
A ← B ← C ← D' ← E' (feature) ↑ (main) # D' and E' are NEW commits! # They have different hashes than D and E # But same changes
5 Fast-forward merge
git checkout main git merge feature # Fast-forward # Result: A ← B ← C ← D' ← E' (main, feature)

Clean, linear history!

✅ When to Use Rebase

  • Local branches: Branches only you have
  • Clean history: Want linear, easy-to-read history
  • Before pull request: Clean up commits before sharing
  • Update feature branch: Bring in latest main changes

Side-by-Side Comparison

Scenario: What Goes Wrong

Result: Teammate gets conflicts, duplicate commits, confusion!

Real-World Workflow

Recommended: Combine Both!

Why this works:

  • Rebase locally: Keep feature branch up to date
  • Merge publicly: Preserve integration points
  • Best of both: Clean commits + preserved history

🤔 What is a "Merge Strategy"?

When you run git merge, Git must decide HOW to combine two branches. The strategy is the algorithm Git uses to figure out what the merged result should look like.

You can pick a strategy with: git merge -s <strategy> branch

Fast-Forward Merge — When No "Real" Merge Is Needed

🤔 What is a Fast-Forward?

When the branch you're merging into has no new commits since the other branch was created, Git doesn't need a merge commit. It just moves the pointer forward.

Fast-Forward vs Three-Way Merge — When Does Each Happen?

⚡ Fast-Forward Merge

Condition: main has NO new commits since feature branched off

Before: A ← B (main) \ C ← D (feature) main is BEHIND feature. No new work on main since B.
git checkout main git merge feature # Updating B..D # Fast-forward ← Git tells you!
After: A ← B ← C ← D (main, feature) # main pointer simply MOVED to D # No merge commit created! # Commits C and D are unchanged
What Git actually did:
1. Read commit D's tree
2. Updated .git/refs/heads/main → D's SHA
3. Updated working directory to match D
That's it! Just moved a pointer.
🔀 Three-Way Merge (ort)

Condition: main HAS new commits since feature branched off

Before: A ← B ← E (main) \ C ← D (feature) Both branches have new work! Can't just move pointer.
git checkout main git merge feature # Merge made by the 'ort' strategy.
After: A ← B ← E ← M (main) \ / C ← D (feature) # M = merge commit with TWO parents # E (main side) and D (feature side)
What Git actually did:
1. Found common ancestor B (merge base)
2. Ran ort: compared B↔E and B↔D
3. Created new tree combining changes
4. Created merge commit M with 2 parents
5. Updated main pointer → M

Controlling Fast-Forward Behavior

# Let Git decide (default): fast-forward if possible, else merge commit git merge feature # Force a merge commit EVEN IF fast-forward is possible git merge --no-ff feature # Why? → preserves the fact that a feature branch existed # ONLY fast-forward, fail if not possible git merge --ff-only feature # Why? → ensures clean linear history, refuses messy merges
git merge feature (default ff)
A ← B ← C ← D (main, feature) # Linear — no merge commit
git merge --no-ff feature
A ← B ←──── M (main) \ / C ← D (feature) # Merge commit M even though # fast-forward was possible!

📊 --no-ff is popular in teams. Why?

Without --no-ff, after deleting a feature branch, you can't tell which commits were part of the feature:

❌ With ff (after branch delete):
A ← B ← C ← D ← E Which commits were the feature? 🤷
✅ With --no-ff (after branch delete):
A ← B ←──── M ← E \ / C ← D Feature commits C,D clearly grouped!

Squash Merge — Flatten Feature Into One Commit

🤔 What is Squash Merge?

Takes ALL commits from a feature branch and combines them into a single commit on the target branch. The feature branch's individual commits are not preserved in the main history.

Squash Merge — Step by Step

Before: Feature has 3 messy commits
A ← B (main) \ C ← D ← E (feature) C = "wip: start login" D = "fix typo" E = "actually finish login"

These messy WIP commits don't belong in main's history!

After: One clean commit on main
A ← B ← S (main) S = "Add login feature" (contains ALL changes from C+D+E) feature branch: C ← D ← E (orphaned)

Clean! One meaningful commit with all the work.

How To Squash Merge

1 Switch to main and run squash merge
git checkout main git merge --squash feature

What Git did:

  • Took ALL changes from C, D, E combined
  • Applied them to working directory + staging area
  • Did NOT create a commit yet!
2 Check what's staged
git status # Changes to be committed: # new file: login.js # modified: app.js # new file: login.css # # All feature changes are staged, ready to commit!
3 Create ONE clean commit
git commit -m "Add login feature" # Result: One commit S on main # S has ONLY ONE parent (B), NOT a merge commit! # Feature branch history is NOT linked
4 Clean up feature branch
git branch -d feature # warning: not fully merged (because squash doesn't link) git branch -D feature # Force delete — that's OK!
Aspect Regular Merge Squash Merge
Commits on main Merge commit + all feature commits visible ONE new commit with combined changes
History link Merge commit has 2 parents → links branches No link → feature branch history is lost
git log on main Shows "wip", "fix typo", etc. Shows only "Add login feature"
git bisect Can find exact commit that broke things Squashed commit is one big change (harder to debug)
Best for Long-lived branches, important history Small features, cleanup of messy WIP commits

⚠️ Squash Merge Gotcha

If you squash-merge a feature branch and then continue working on it, you'll get conflicts when merging again! Why?

# Bad pattern: git merge --squash feature # Creates S on main git commit -m "Add login" # ... later, more work on feature branch ... git merge --squash feature # 💥 CONFLICTS! # Git doesn't know S already contains C,D,E's changes # because S has no link to feature branch history

Rule: After squash merge, delete the feature branch and create a new one if more work is needed.

Cherry-Pick — Take Specific Commits From Another Branch

🍒 What is Cherry-Pick?

Cherry-pick takes one specific commit from any branch and replays its changes on your current branch. It creates a new commit with the same changes but a different SHA.

Analogy: It's like photocopying one page from someone else's notebook into yours.

Cherry-Pick — Visual Flow

Before: You need commit D's fix on main
A ← B ← E (main) ← HEAD \ C ← D ← F (feature) ↑ This commit fixes a critical bug! But feature branch isn't ready to merge yet.
git checkout main git cherry-pick D # use D's actual SHA
After: D's changes copied onto main as new commit D'
A ← B ← E ← D' (main) ← HEAD \ C ← D ← F (feature) D' has the SAME changes as D but a DIFFERENT SHA (different parent!)

Cherry-Pick Commands

# Cherry-pick one commit git cherry-pick c00002 # Cherry-pick multiple commits git cherry-pick c00001 c00002 c00003 # Cherry-pick a range (exclusive start, inclusive end) git cherry-pick c00001..c00003 # picks c00002 and c00003 # Cherry-pick WITHOUT auto-committing (stage changes only) git cherry-pick --no-commit c00002 # → changes are staged but not committed # → useful when you want to combine with other changes # If there's a conflict during cherry-pick: git cherry-pick c00002 # CONFLICT (content): Merge conflict in hello.txt # Fix the conflict, then: git add hello.txt git cherry-pick --continue # Or abort the cherry-pick entirely: git cherry-pick --abort

🔬 Cherry-Pick at the Object Level

What Git actually does when you cherry-pick commit D:

1 Calculate the diff
Git computes: D minus D's parent (C) → gets the patch (what D changed)
# D changed hello.txt: # - "Hello World!" # + "Hello World!\nBug fixed!"
2 Apply patch to current HEAD
Git applies that patch to main's current state (commit E)
3 Create new objects
  • 🆕 New blob for modified hello.txt
  • 🆕 New tree pointing to new blob + reused blobs
  • 🆕 New commit D' — parent is E (not C!), different SHA
4 Result
D' has same changes as D, but is a completely different object (different parent → different SHA)

🍒 When to Use Cherry-Pick

✅ Good Use Cases
  • Hotfix: Bug fix on feature branch needed on main NOW
  • Backport: Apply fix from main to older release branch
  • Selective merge: Only want specific commits, not whole branch
  • Recover commit: Rescue a commit from a deleted branch
❌ Avoid When
  • Merging entire branch: Use merge or rebase instead
  • Frequent cherry-picking: Sign of bad branching strategy
  • Duplicate commits: D and D' have same changes — if you later merge feature into main, Git may show conflicts

Merge vs Squash vs Rebase vs Cherry-Pick — Decision Guide

Method What Happens History Shape Best For
merge Creates merge commit with 2 parents. All commits preserved. ─╲─╱─ (diamond) Team branches, preserving complete history
merge --no-ff Same as merge, but FORCES merge commit even if ff possible ─╲─╱─ (always) Teams wanting clear feature grouping
merge --squash Flattens all commits into one. No branch link. ───── (linear, 1 commit) Small features, cleaning up messy WIP commits
rebase + merge Replays commits on top, then fast-forwards. Clean linear history. ───── (linear, all commits) Personal branches, clean history, before PR
cherry-pick Copies one specific commit to current branch. ─ · ─ (one commit) Hotfixes, backports, selective commits

⏮️8. Undoing Changes — What to Use When

Three Ways to Undo, Three Different Use Cases

Understanding WHEN to use each command is crucial!

git restore - Undo Working Directory Changes

Use When: You want to discard uncommitted changes

Scenario: You messed up a file and want to revert to last commit
# You modified a file echo "bad changes" >> important.txt git status # Changes not staged: # modified: important.txt # Discard changes git restore important.txt # File is back to last committed state! git status # nothing to commit, working tree clean

What happened:

  • Git looked at HEAD commit
  • Found the blob for important.txt
  • Overwrote working directory file with blob content
  • Changes are PERMANENTLY LOST!
Advanced: Restore from staging area
# Staged a file git add file.txt # Made more changes echo "more" >> file.txt # Restore to staged version (not last commit) git restore file.txt # Gets version from index

git reset - Move Branch Pointer

Use When: You want to undo commits or unstage files

⚠️ Three Modes of Reset
  • --soft: Move HEAD, keep staging area and working directory
  • --mixed (default): Move HEAD, update staging area, keep working directory
  • --hard: Move HEAD, update staging area AND working directory
Mode 1: git reset --soft
Before
A ← B ← C ← D (main) ↑ HEAD Working dir: clean Staging: empty
After: git reset --soft HEAD~1
A ← B ← C (main) ↑ HEAD Working dir: changes from D Staging: changes from D

Use case: "I want to redo my last commit with different message"

git reset --soft HEAD~1 # Now everything is staged, ready to commit again git commit -m "Better commit message"
Mode 2: git reset --mixed (default)
Before
A ← B ← C ← D (main) ↑ HEAD Working dir: clean Staging: empty
After: git reset HEAD~1
A ← B ← C (main) ↑ HEAD Working dir: changes from D Staging: empty (unstaged!)

Use case: "I want to undo commit but keep my changes to edit"

git reset HEAD~1 # Changes now in working directory # Can split into multiple commits git add part1.txt git commit -m "Part 1" git add part2.txt git commit -m "Part 2"
Mode 3: git reset --hard
Before
A ← B ← C ← D (main) ↑ HEAD Working dir: clean Staging: empty
After: git reset --hard HEAD~1
A ← B ← C (main) ↑ HEAD Working dir: clean (matches C) Staging: empty

Use case: "I want to completely discard last commit(s)"

git reset --hard HEAD~1 # Everything from commit D is GONE! # Working directory matches commit C
⚠️ DANGER: Changes are permanently lost!

Exception: Commits still in reflog for ~30 days

# Recover lost commit git reflog # c00002 HEAD@{1}: reset: moving to HEAD~1 # c00003 HEAD@{2}: commit: Lost commit git checkout c00003 # Recovered!
Unstaging Files
# Staged files you don't want to commit git add . git status # Changes to be committed: # modified: hello.txt # modified: app.js # Unstage one file git reset HEAD hello.txt # Now only app.js is staged git status # Changes to be committed: # modified: app.js # Changes not staged: # modified: hello.txt

git revert - Create Inverse Commit

Use When: You want to undo a commit but keep history (safe for public branches)

Scenario: Bad commit was pushed to shared branch
1 Initial state
A ← B ← C ← D (main) ↑ HEAD # Commit D has a bug!
2 Revert commit D
git revert D # Git creates new commit that undoes D
3 Result
A ← B ← C ← D ← E (main) ↑ HEAD # E undoes everything D did # History is preserved! # Safe to push!

What's in commit E:

git log -1 --format=fuller # Revert "Bad commit message from D" # This reverts commit c00003... # The tree in E matches the tree in C # But history shows D happened and was reverted

✅ Why Revert is Safe for Public Branches

  • Doesn't rewrite history (no force push needed)
  • Creates new commit (everyone can pull normally)
  • Shows what happened and when
  • Can be reverted again if needed
Revert Multiple Commits
# Revert last 3 commits git revert HEAD~2..HEAD # Or revert a range git revert c00001..c00003 # Revert merge commit (need -m flag) git revert -m 1 merge_commit_hash

🌐9. Remote Operations — push, pull, fetch, clone

"origin" is Just a Name!

There's nothing special about "origin". It's just the default name Git gives to the remote you cloned from.

Understanding Remotes

# Add a remote git remote add origin https://github.com/user/repo.git # See what happened cat .git/config [remote "origin"] url = https://github.com/user/repo.git fetch = +refs/heads/*:refs/remotes/origin/* # "origin" is just stored in config! # You can call it anything: git remote add backup https://gitlab.com/user/repo.git git remote add upstream https://github.com/original/repo.git

git fetch - Download Without Merging

What git fetch Does

1 Initial state
Local: A ← B ← C (main) ↑ HEAD Remote (origin): A ← B ← C ← D ← E (origin/main) # Remote is 2 commits ahead!
2 Run git fetch
git fetch origin

Behind the scenes:

  • Connect to remote repository
  • Download commits D and E
  • Download referenced trees and blobs
  • Update refs/remotes/origin/main
  • DON'T touch your local branch!
3 After fetch
A ← B ← C (main) ← Your branch (unchanged) ↑ HEAD A ← B ← C ← D ← E (origin/main) ← Remote tracking branch # Commits downloaded but NOT merged!
4 Inspect what was fetched
# See what's new git log main..origin/main # commit E # commit D # Your code still at C ls # Files match commit C # Merge when ready git merge origin/main # Or git rebase origin/main

✅ Why Use Fetch?

  • Safe: Doesn't change your working code
  • Inspect first: See what changed before merging
  • Choose merge strategy: Merge or rebase
  • Multiple remotes: Fetch from several sources

git pull - Fetch + Merge Combined

What git pull Does

1 Run git pull
git pull origin main # Equivalent to: git fetch origin git merge origin/main
2 What happens
# Before: A ← B ← C (main, HEAD) A ← B ← C ← D ← E (origin/main) # After pull: A ← B ← C ← D ← E (main, HEAD, origin/main)

⚠️ Pull with Diverged Branches

# Local has commit F, remote has commits D and E A ← B ← C ← F (main, HEAD) A ← B ← C ← D ← E (origin/main) # git pull creates merge commit: A ← B ← C ← F ← M (main, HEAD) \ / D ← E (origin/main)

Alternative: Pull with rebase

git pull --rebase origin main # Result: Linear history A ← B ← C ← D ← E ← F' (main, HEAD)

git push - Upload Commits

What git push Does

1 Local commits
Local: A ← B ← C ← D ← E (main, HEAD) Remote: A ← B ← C (origin/main) # You're 2 commits ahead
2 Push commits
git push origin main

Behind the scenes:

  • Git checks if remote has commits you don't have
  • If yes: REJECT (must pull first)
  • If no: Upload commits D and E
  • Upload referenced trees and blobs
  • Update remote branch pointer
  • Update local tracking branch
3 After push
Local: A ← B ← C ← D ← E (main, HEAD, origin/main) Remote: A ← B ← C ← D ← E (main) # Everything in sync!

⚠️ Push Rejected

git push origin main ! [rejected] main -> main (fetch first) error: failed to push some refs # Why? Remote has commits you don't have! # Fix: git pull origin main # Fetch and merge # Resolve conflicts if any git push origin main # Try again

🔥 Force Push - DANGEROUS!

git push --force origin main # ⚠️ OVERWRITES remote history! # Use ONLY if: # 1. You're absolutely sure # 2. It's your branch only # 3. You understand consequences # Safer: Use --force-with-lease git push --force-with-lease origin main # Rejects if someone else pushed

Tracking Branches

What are tracking branches?

Local branches that have a direct relationship to a remote branch.

# Create tracking branch git checkout -b feature origin/feature # Or git checkout --track origin/feature # Now git push/pull knows where to go! git push # Automatically pushes to origin/feature git pull # Automatically pulls from origin/feature # See tracking relationships git branch -vv # main c00002 [origin/main: ahead 2] Add welcome # feature c00003 [origin/feature] Add styling

"ahead 2" means you have 2 commits not yet pushed.

"behind 1" means remote has 1 commit you don't have.

🔒10. Security — Removing Sensitive Data

⚠️ You Committed Secrets - Now What?

Critical: Just deleting the file and committing DOES NOT REMOVE IT FROM HISTORY!

❌ Why This Doesn't Work

# Accidentally committed credentials git add secrets.txt git commit -m "Add config" git push # Oh no! Delete it git rm secrets.txt git commit -m "Remove secrets" git push # ❌ SECRETS STILL IN HISTORY! git log --all -- secrets.txt # Still there! git checkout HEAD~1 -- secrets.txt # Can recover it!

Anyone who cloned your repo has the secrets!

✅ Correct Way: Rewrite History

Method 1: BFG Repo-Cleaner (Recommended)

1 Download BFG
# Download from: https://rtyley.github.io/bfg-repo-cleaner/ # Or with brew: brew install bfg
2 Delete file from history
bfg --delete-files secrets.txt # Or delete by content bfg --replace-text passwords.txt # Replace matches with ***REMOVED***

What BFG does:

  • Rewrites every commit
  • Removes the file/content
  • Updates all tree objects
  • Creates new commit hashes
3 Clean up and push
# Remove old references git reflog expire --expire=now --all # Garbage collect git gc --prune=now --aggressive # Force push git push origin --force --all git push origin --force --tags
4 CRITICAL: Rotate credentials!
# Change ALL exposed credentials: # - API keys # - Passwords # - Tokens # - Certificates # Assume they're compromised! # Even after removing from Git, they may be cached/cloned

Method 2: git filter-branch (Built-in)

1 Remove file from all commits
git filter-branch --force --index-filter \ 'git rm --cached --ignore-unmatch secrets.txt' \ --prune-empty --tag-name-filter cat -- --all
2 Clean and push
git reflog expire --expire=now --all git gc --prune=now --aggressive git push origin --force --all
Note: filter-branch is deprecated. Use BFG or git-filter-repo instead.

Advanced Concepts

Why Git Doesn't Track Empty Folders

Reason: Git only tracks file content (blobs). Folders only exist as entries in tree objects.

The Problem

The Solution

# Convention: Add .gitkeep (empty file) touch empty-folder/.gitkeep git add empty-folder/.gitkeep git commit -m "Preserve folder structure" # Now Git creates: # - Blob (empty .gitkeep file) # - Tree (empty-folder directory) # - Commit (referencing tree)

The Reflog - Your Safety Net

What is Reflog?

Git keeps a log of every time HEAD moves. It's your undo button for Git itself!

git reflog # c00002 HEAD@{0}: commit: Add welcome # c00001 HEAD@{1}: commit: First commit # c00003 HEAD@{2}: reset: moving to HEAD~1 # c00004 HEAD@{3}: commit: Lost commit # Recover "lost" commit! git checkout c00004 git branch recovered-work # Save it! git checkout main git merge recovered-work

✅ Reflog Saves You From:

  • Accidental git reset --hard
  • Deleted branches
  • Bad rebases
  • Mistaken merges

Reflog entries last ~30-90 days

🏷️12. Tags — Version Markers

Tags: Bookmarks in Git History

Mark important points in your project's history

🤔 What are Tags?

Simple Answer: Tags are permanent bookmarks to specific commits. Unlike branches that move forward with new commits, tags stay at one commit forever.

Real-World Analogy: Imagine a book where branches are sticky notes you keep moving as you read, but tags are permanent page numbers you write in the index for important chapters.

Why Use Tags?

Branching vs Tagging

Branch (moves with commits):
                              main
                               ↓
A ← B ← C ← D ← E ← F ← G ← [H]
                        ↑
                      develop

                                   main
                                    ↓
A ← B ← C ← D ← E ← F ← G ← H ← [I]  ← Branch moved!
                        ↑
                      develop

        v1.0         v2.0

👉13. HEAD vs Branch Pointers

Understanding Git's Pointer System

HEAD and branches are just files with commit hashes!

🤔 What are Pointers in Git?

Simple Answer: Pointers are just files containing commit hashes. They "point" to commits by storing their SHA-1 hash.

Real-World Analogy: Think of Git's history as a library of books (commits). Pointers are bookmarks that tell you which book to read right now.

Branch Pointers

# A branch is just a file! cat .git/refs/heads/main c00002... # That's literally it. Just a 40-character hash.

How Branches Work

File System:
.git/refs/heads/main      → c00002...
.git/refs/heads/develop   → c00001...
.git/refs/heads/feature   → c00003...

                                      main

A ← B ← C ← D ← E ← F ← c00002 ← [NEW]

# HEAD is also just a file! cat .git/HEAD ref: refs/heads/main # It usually contains a REFERENCE to a branch # This is called "attached HEAD"

File: .git/HEAD
ref: refs/heads/main

Meaning: HEAD points to "main" branch, which points to a commit

Detached HEAD State

Simple: HEAD points directly to a commit, not to a branch.

Why it matters: Commits you make won't belong to any branch!

Detached HEAD - Visual

  • Branches are just files containing commit hashes in .git/refs/heads/

❌ Error 1: "fatal: not a git repository"

🔴 The Error
$ git status fatal: not a git repository (or any of its parent directories): .git

Plain English: You're in a folder that doesn't have a .git directory. Git doesn't know this folder.

✅ The Fix

❌ Error 2: "error: failed to push some refs"

🔴 The Error
$ git push origin main ! [rejected] main -> main (non-fast-forward) error: failed to push some refs to 'https://github.com/user/repo.git' hint: Updates were rejected because the remote contains work that you do not have locally.

Plain English: Someone pushed changes before you. The remote is ahead. Git won't let you overwrite their work.

✅ The Fix
# Step 1: Pull remote changes first git pull origin main # (fix any merge conflicts if they happen) # Step 2: Push your combined changes git push origin main # Alternative: Rebase instead of merge git pull --rebase origin main git push origin main
Visual:
Remote: A ← B ← C ← D (someone else's commit) Local: A ← B ← C ← E (your commit) After pull: A ← B ← C ← D ← E ← M (merge) Now push works! ✅

❌ Error 3: "CONFLICT (content): Merge conflict in file.txt"

🔴 The Error
$ git merge feature Auto-merging hello.txt CONFLICT (content): Merge conflict in hello.txt Automatic merge failed; fix conflicts and then commit the result.

Plain English: Both branches changed the same lines in the same file. Git can't decide which version to keep.

✅ The Fix
# 1. Open the file — you'll see markers: <<<<<<< HEAD Hello from main ======= Hello from feature >>>>>>> feature # 2. Edit to keep what you want: Hello from main and feature # 3. Remove ALL markers, then: git add hello.txt git commit -m "Resolve merge conflict"

👉 See Section 16 below for the complete merge conflict deep dive!

❌ Error 4: "error: Your local changes would be overwritten by merge/checkout"

🔴 The Error
$ git checkout feature error: Your local changes to the following files would be overwritten by checkout: hello.txt Please commit your changes or stash them before you switch branches.

Plain English: You have unsaved/uncommitted changes in files that the target branch also modifies. Git refuses to lose your work.

✅ The Fix (3 options)
# Option 1: Stash your changes (save for later) git stash git checkout feature # ... do work on feature ... git checkout main git stash pop # Restore your changes # Option 2: Commit your changes first git add . git commit -m "WIP: save progress" git checkout feature # Option 3: Discard changes (DANGEROUS!) git checkout -f feature # Loses changes!

❌ Error 5: "detached HEAD state"

🔴 The Warning
$ git checkout c00001 Note: switching to 'c00001'. You are in 'detached HEAD' state. HEAD is now at c00001 Initial commit

Plain English: HEAD points directly at a commit, NOT at a branch. Any new commits you make won't belong to any branch and may get lost!

✅ The Fix
# If you just want to look around: git checkout main # Go back to branch # If you made commits in detached HEAD: git branch rescue-branch # Save them! git checkout main git merge rescue-branch # Prevent detached HEAD in the future: # Use branches, not raw commit SHAs
Normal: HEAD → main → c00003 Detached: HEAD → c00001 (no branch!) Commits made in detached HEAD are "orphaned" — GC will delete them!

❌ Error 6: "fatal: refusing to merge unrelated histories"

🔴 The Error
$ git pull origin main fatal: refusing to merge unrelated histories

Plain English: Your local repo and the remote have NO common ancestor commit. They're completely separate histories (e.g., you did git init locally AND created a new repo on GitHub with a README).

✅ The Fix
# Allow merging unrelated histories git pull origin main --allow-unrelated-histories # Fix any conflicts, then commit git add . git commit -m "Merge remote into local" git push origin main

❌ Error 7: "fatal: The current branch has no upstream branch"

🔴 The Error

Plain English: You created this branch locally. Git doesn't know which remote branch to push to because you never linked them.

✅ The Fix
# Set upstream and push (first time only) git push -u origin feature # -u = --set-upstream # After this, just use: git push # Git remembers the link! # What changed in .git/config: [branch "feature"] remote = origin merge = refs/heads/feature

❌ Error 8: "error: pathspec 'branch-name' did not match any file(s)"

🔴 The Error

Plain English: That branch doesn't exist locally. Maybe it's a remote branch you haven't fetched, or a typo.

✅ The Fix

❌ Error 9: "Undo last commit" (Not an error, but most searched!)

The 3 Levels of "Undo Commit"

As if the commit never happened. # Level 4: Already pushed? Create a REVERSE commit git revert HEAD # Creates NEW commit that undoes the last one # Safe for shared branches!

⚔️16. Merge Conflicts — Complete Deep Dive

Why Do Merge Conflicts Happen?

A merge conflict happens when Git cannot automatically decide which changes to keep. This ONLY happens when two branches modify the same lines in the same file.

When Does Git Conflict vs Auto-Merge?

✅ NO Conflict — Auto-Merged
Scenario 1: Different files changed main changed: hello.txt feature changed: app.js → Git merges both ✅ Scenario 2: Same file, different lines main changed: line 5 feature changed: line 20 → Git merges both ✅ Scenario 3: One side changed, other didn't main changed: line 10 feature didn't touch line 10 → Git takes the change ✅
❌ CONFLICT — Manual Fix Needed
Scenario 4: Same lines changed differently main line 5: "Hello World" feature line 5: "Hello Git" → 💥 CONFLICT! Scenario 5: One deleted, other modified main: deleted hello.txt feature: modified hello.txt → 💥 CONFLICT! Scenario 6: Both added same filename main: created config.json feature: created config.json → 💥 CONFLICT!

Anatomy of a Conflict — What You See in the File

Understanding Conflict Markers

Line 1: This line is the same in both branches
Line 2: This line is also the same

<<<<<<< HEAD ← YOUR branch (current/main)
Hello from main branch
We added this greeting on main
======= ← DIVIDER between the two versions
Hello from feature branch
We wrote a different greeting here
>>>>>>> feature ← THEIR branch (incoming/feature)

Line 7: This line is the same in both branches
⬆️
<<<<<<< HEAD

Start of YOUR changes (current branch)

➖
=======

Divider between the two versions

⬇️
>>>>>>> feature

Resolving a Conflict — Step by Step

1 See which files have conflicts
git status # Unmerged paths: # both modified: hello.txt # both modified: app.js
2 Open each conflicted file and decide what to keep
Option A: Keep Ours
Hello from main branch We added this greeting on main
Option B: Keep Theirs
Hello from feature branch We wrote a different greeting here
Option C: Combine Both

3 Remove ALL conflict markers
# Delete these lines completely: <<<<<<< HEAD ← DELETE this line ======= ← DELETE this line >>>>>>> feature ← DELETE this line # Keep ONLY the content you want
4 Mark as resolved and commit
git add hello.txt app.js git commit # Git auto-creates merge message: # "Merge branch 'feature' into main" # Or with custom message: git commit -m "Merge feature: resolve greeting conflict"

The 8 Golden Rules for Conflict-Free Git

✅ Rule 1: Pull Before You Start Working

❌ Without Rule 1 — What Goes Wrong

What Git does when you git pull

✅ Rule 2: Pull Frequently While Working

❌ Sync Once After 5 Days
Day 1: Branch created from main main: A ← B feature: A ← B Day 2-5: Both branches growing separately main: A ← B ← C ← D ← E ← F ← G feature: A ← B ← H ← I ← J ← K ← L Day 5 merge attempt: 5 main commits × 5 feature commits = HUGE diff to reconcile = 💥 12 conflicting files!

Pain level: 🔥🔥🔥🔥🔥
Massive diff, hard to understand what changed where. Takes hours to resolve.

✅ Sync Every Day
Day 1: git merge main → +1 commit feature: A ← B ← H ← M₁ (merge C) Day 2: git merge main → +1 commit feature: ...← M₁ ← I ← M₂ (merge D,E) Day 3: git merge main → +1 commit feature: ...← M₂ ← J ← M₃ (merge F) Each merge is TINY: Only 1-2 new commits from main each time = easy auto-merge or 1 small conflict Day 5 final merge to main: Already up-to-date with main! = ✅ Clean merge, 0-1 conflicts

Pain level: 😊
Small incremental merges. Each takes seconds. Conflicts caught when they're tiny.

Daily sync workflow

Creates merge commit with 2 parents # Your branch now has ALL of main's latest changes! # OPTION B: Rebase (cleaner history, rewrites commits) git checkout feature/my-work git fetch origin git rebase origin/main # Git replays your commits ONE BY ONE on top of latest main # Each commit is re-applied → if conflict, you fix it for THAT commit # Result: linear history, as if you started from latest main # WHEN to use which: # merge → shared branches, preserving history # rebase → personal branches, clean linear history

✅ Rule 3: Keep Branches Short-Lived

❌ 3-Week Feature Branch
Week 1: 8 commits on main, 5 on feature Week 2: 12 commits on main, 8 on feature Week 3: 6 commits on main, 4 on feature main: 26 new commits ──────────→ feature: 17 new commits ──────────→ Divergence: 43 commits apart! Files changed on main: 35 Files changed on feature: 22 Files changed on BOTH: 11 💥 Merge result: 11 potential conflicts Time to merge: half a day
✅ 2-Day Feature Branch
Day 1: 4 commits on main, 3 on feature Day 2: 3 commits on main, 2 on feature main: 7 new commits ──→ feature: 5 new commits ──→ Divergence: 12 commits apart Files changed on main: 8 Files changed on feature: 4 Files changed on BOTH: 1 Merge result: 0-1 conflicts Time to merge: 30 seconds
🔬 Why This Works — The Math of Divergence

Conflict probability formula (simplified):

P(conflict) ≈ files_changed_main × files_changed_feature / total_files²

3-week branch:
35 × 22 / 100² = 7.7% per file
With 100 files → ~7-8 conflicts expected
2-day branch:
8 × 4 / 100² = 0.32% per file
With 100 files → ~0 conflicts expected

Real-world rule: If your branch lives longer than 3 days, you're asking for trouble. Break big features into smaller PRs.

✅ Rule 7: Consistent Code Formatting

❌ Without Formatting Rules
Alice uses tabs, Bob uses 2 spaces: Alice's version (hello.txt): function greet() { → console.log('hello'); ← TAB → return true; ← TAB } Bob's version (hello.txt): function greet() { console.log('hello'); ← 2 SPACES return true; ← 2 SPACES } Git sees EVERY line as changed! ✅ With Formatting Rules

Git only sees REAL logic differences. No whitespace noise. git merge → ✅ Only real conflicts if any

Only REAL code changes show up in diffs — no whitespace noise.

📦17. Git LFS — Large File Storage In Detail

The Problem: Git Wasn't Built for Large Files

Git stores every version of every file as a blob. A 100MB video file with 10 versions = 1GB in .git! Even with packfiles and delta compression, binary files (images, videos, datasets, models) don't compress well.

Git LFS (Large File Storage) solves this by storing large files on a separate server and keeping only tiny pointer files in your Git repo.

How Git LFS Works — The Big Picture

❌ Without Git LFS
.git/objects/ ├── blob (hello.txt) 2 KB ├── blob (app.js) 5 KB ├── blob (style.css) 3 KB ├── blob (video.mp4 v1) 100 MB ├── blob (video.mp4 v2) 102 MB ├── blob (video.mp4 v3) 98 MB ├── blob (model.bin v1) 500 MB ├── blob (model.bin v2) 510 MB └── trees, commits... ~1 KB Total .git size: ~1.3 GB 😱 git clone takes 20 minutes!
→
✅ With Git LFS
.git/objects/ (normal Git) ├── blob (hello.txt) 2 KB ├── blob (app.js) 5 KB ├── blob (style.css) 3 KB ├── pointer (video.mp4) ~130 bytes ├── pointer (model.bin) ~130 bytes └── trees, commits... ~1 KB LFS Server (separate!): ├── video.mp4 v1 100 MB ├── video.mp4 v2 102 MB ├── video.mp4 v3 98 MB ├── model.bin v1 500 MB └── model.bin v2 510 MB .git size: ~11 KB 🎉 git clone takes 2 seconds! (LFS files downloaded on-demand)

What's Inside an LFS Pointer File?

What Git stores (pointer file):
# This is ALL that's in your .git for video.mp4: version https://git-lfs.github.com/spec/v1 oid sha256:4d7a214614ab2935c943f9... size 104857600

Just 3 lines! ~130 bytes instead of 100MB.

Pointer file decoded:
  • version: LFS protocol version (always v1)
  • oid: SHA-256 hash of the actual file content — this is how LFS finds the real file on the server
  • size: File size in bytes (so Git knows how big it'll be when downloaded)

When you git checkout, the LFS smudge filter replaces this pointer with the actual 100MB file in your working directory.

🔬 How LFS Works Under the Hood — The Smudge/Clean Filter

🧹 Clean Filter (git add / commit)
You run: git add video.mp4 1. Git calls LFS "clean" filter 2. LFS reads the 100MB file 3. LFS computes SHA-256 hash 4. LFS stores file in .git/lfs/objects/ 5. LFS returns a 130-byte POINTER 6. Git stores the POINTER as a blob 7. Commit contains pointer, not file!
💧 Smudge Filter (git checkout / pull)
You run: git checkout main 1. Git reads pointer blob from commit 2. Git calls LFS "smudge" filter 3. LFS reads the pointer (SHA-256 hash) 4. LFS checks .git/lfs/objects/ cache 5. If cached → use local copy If not → download from LFS server 6. LFS replaces pointer with REAL file 7. Working dir has actual 100MB video!

14 Create a conflict on purpose
# Create a new branch and change line 1 of hello.txt git checkout -b feature/new-header sed -i 's/Hello World!/Hello Universe!/' hello.txt git add . && git commit -m "Change to Universe" # Switch back to main and change the SAME line git checkout main sed -i 's/Hello World!/Hello Planet!/' hello.txt git add . && git commit -m "Change to Planet" # Now merge — CONFLICT! git merge feature/new-header

<<<<<<< HEAD Hello Planet! ======= Hello Universe! >>>>>>> feature/new-header Welcome to hello-app! Greetings from the feature branch!

# Edit hello.txt to keep both: cat > hello.txt << 'EOF' Hello Planet and Universe! Welcome to hello-app! Greetings from the feature branch! EOF # Mark as resolved and commit git add hello.txt git commit -m "Resolve header conflict: combine Planet + Universe"

⛔14. When NOT to Rebase — Critical Rules

The Golden Rule of Rebase

NEVER rebase commits that exist outside your repository!

🚨 DANGER ZONE: When Rebase Goes Wrong

Layman Explanation: Once you've shared your commits with others (pushed to shared branch), rebasing rewrites history. This creates duplicate commits and confuses everyone.

Scenario: You rebase a public branch

Before (everyone has this):
A ← B ← C ← D ← E  (main, on server)
         ↑
    Everyone pulled this

You locally rebase:
A ← B ← C' ← D' ← E'  (your main)
         ↑
    New hashes! Different commits!

You force push:
Server now has: A ← B ← C' ← D' ← E'

Others still have:
Their local: A ← B ← C ← D ← E

When they pull:
     C ← D ← E  ← Old commits (they still have)
    ↗         ↖
A ← B          ← MERGE COMMIT?!
    ↖         ↗
     C' ← D' ← E'  ← New commits from server

Result: DUPLICATE COMMITS! C, D, E appear twice!
Everyone gets merge conflicts!

Rules: When to Rebase vs Merge

Situation Use Why
Local feature branch ✅ REBASE Clean history, only you have these commits
Shared feature branch (team working) ❌ MERGE Others have these commits, rebase will duplicate
Main/master branch ❌ NEVER REBASE Everyone pulls from it, rewriting causes chaos
Updating your branch with main ✅ REBASE Keeps your branch linear and up-to-date
Pull Request / Merge Request ⚠️ DEPENDS Rebase before opening PR (clean history). After PR open, avoid rebase if others reviewing
Already pushed to origin ❌ NO REBASE Unless it's YOUR feature branch and you coordinate with team
Cleaning up messy local commits ✅ REBASE -i Interactive rebase is perfect for this!

Safe Rebase Workflow

Scenario: Update Your Feature Branch

# You're working on feature branch (never pushed OR only you use it) git checkout feature/user-auth # Main has new commits, you want them git fetch origin main git rebase origin/main # If conflicts, resolve and continue git add resolvedFile.js git rebase --continue # Push to YOUR feature branch # If first push: git push -u origin feature/user-auth # If you already pushed and rebased (coordinate with team first!): git push --force-with-lease origin feature/user-auth

⚠️ Force Push with Lease (Safer)

# BAD (can overwrite others' work): git push --force # GOOD (checks if someone else pushed): git push --force-with-lease # force-with-lease fails if remote has commits you don't have # This prevents accidentally destroying others' work!

✅ Safe Rebase Checklist

  • ✅ Branch is local only (never pushed)
  • ✅ OR branch is your personal feature branch
  • ✅ No one else is working on this branch
  • ✅ Branch is NOT main/master/develop
  • ✅ If already pushed, coordinated with team
  • ✅ Using --force-with-lease not --force

💡 Real-World Examples

Example 1: Solo Developer (Safe to Rebase)
# You work alone on feature branch git checkout feature/new-dashboard # Make commits... git commit -m "WIP: dashboard" git commit -m "More work" git commit -m "Fix typo" # Rebase to clean up before merging to main git rebase -i HEAD~3 # Squash commits into one # Result: One clean commit # Push (force needed because you rewrote history) git push --force-with-lease origin feature/new-dashboard ✅ SAFE: Only you use this branch
Example 2: Team Branch (NOT Safe to Rebase)
# You and colleague both work on feature/big-feature # Colleague pushed commits C, D # You have local commits E, F git log origin/feature/big-feature A ← B ← C ← D (Colleague's work) git log feature/big-feature A ← B ← C ← D ← E ← F (Your work) # DON'T rebase main into this! ❌ git rebase main # Will rewrite C, D, E, F # Colleague still has old C, D → CONFLICT! # Instead, merge: ✅ git merge main # Creates merge commit # Everyone sees same history
